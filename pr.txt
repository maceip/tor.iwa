## Title

Add Tor Hidden Service IWA — WebMCP tools for anonymous AI agent networking with OHTTP relay mesh

## Body

### Summary

This PR adds a **Tor Hidden Service running inside an Isolated Web App** with 7 WebMCP tools that give AI agents capabilities they can't get any other way: anonymous `.onion` browsing with real OHTTP privacy via a peer relay mesh, peer-to-peer connections over Tor, and cryptographic access control for hidden services.

**What's new:** Every tor.iwa instance can **volunteer as an OHTTP relay** for other instances. When an agent calls `fetchOnion` with `useOHTTP=true`, the request is ECDH-encrypted and sent to a peer's `.onion`, where the peer decrypts and fetches the target on its own Tor circuit. This provides real application-layer unlinkability — the target sees the relay's circuit, not the requester's. When no peers are available, the system automatically falls back to **Tor circuit isolation** (unique SOCKS5 credentials force a separate circuit via `IsolateSOCKSAuth`).

### Why this is interesting for WebMCP

This is the first WebMCP integration that gives AI agents access to **network-level privacy infrastructure** rather than just application features:

1. **`fetchOnion`** — AI agents cannot normally reach `.onion` addresses. This tool routes requests through a real Tor circuit via Direct Sockets SOCKS5. With `useOHTTP=true`, requests are encrypted and relayed through a peer IWA instance for application-layer unlinkability, or use circuit isolation as an automatic fallback. Returns HTTP response + TOFU cert verification + OHTTP mode used.

2. **`manageOHTTPRelay`** — The relay mesh management tool. `volunteer` makes this instance accept OHTTP relay requests from peers. `addPeer`/`removePeer` manages known relays. `status` shows relay stats. This creates a decentralized privacy network where every tor.iwa user is a potential relay for every other user.

3. **`holepunch`** — Opens real bidirectional TCP connections to other `.onion` hidden services. Foundation for **anonymous AI-to-AI communication** — Agent A publishes a `.onion`, Agent B holepunches to it, they exchange messages without either knowing the other's IP. Supports `connect` / `send` / `receive` / `close` actions.

4. **`validateOnionCert`** — Trust-on-first-use (TOFU) certificate management. Service fingerprints are auto-captured on every `fetchOnion` call. Detects identity changes that may indicate service compromise.

5. **`manageTrustedClients`** — Real access control enforcement on the hidden service. Rejects requests without a valid `X-Client-ID` header (returns 403). Supports Ed25519 public key registration and signature verification via `X-Client-Auth` headers.

6. **`getServiceStatus`** — Aggregated dashboard including OHTTP relay state: volunteering status, peer count, requests relayed, bytes relayed, plus HS uptime/stats, cert store health, and active holepunch sessions.

7. **`listHolepunchSessions`** — Lists all peer connections with connection status, message/byte counts, and timestamps.

### OHTTP relay architecture

```
┌──────────┐    ECDH-encrypted    ┌──────────────┐    Separate     ┌──────────┐
│ Your IWA │ ────────────────────→ │ Peer Relay   │ ──────────────→ │  Target  │
│          │    via Tor circuit A  │ (.onion IWA) │  Tor circuit B  │  .onion  │
└──────────┘                      └──────────────┘                 └──────────┘
```

**How it works:**
1. Sender encodes request as BHTTP (RFC 9292), encrypts with peer's ECDH P-256 public key via AES-GCM
2. Sends encrypted envelope to `peer.onion/.well-known/ohttp-relay` through Tor circuit A
3. Peer relay decrypts using ECDH key agreement, extracts BHTTP request
4. Fetches target on its own Tor circuit B (different guard/middle/exit)
5. Encrypts response back via same ECDH shared secret

**Privacy guarantees:**
- Tor hides requester's IP from relay (circuit A)
- Target sees circuit B, not circuit A — can't link request to requester
- AES-GCM encryption prevents Tor exit nodes from reading the OHTTP envelope
- Each request uses a fresh ephemeral ECDH key pair

**Fallback (no peers):**
- Uses SOCKS5 username/password auth with unique random credentials per request
- Tor's `IsolateSOCKSAuth` forces a fresh circuit per request
- Different guard/middle/exit nodes than regular traffic

### Technical highlights

- **No build step** — Pure ESM with Preact + htm, vendored dependencies
- **Real Tor** — `tor.js` + `tor.wasm` compiled via Emscripten, connects to the actual Tor network
- **Real TCP** — `TCPSocket` for SOCKS5 outbound, `TCPServerSocket` for HS inbound (Direct Sockets API)
- **Real OHTTP privacy** — ECDH P-256 key exchange, AES-GCM encryption, BHTTP framing, peer relay over Tor
- **Circuit isolation fallback** — SOCKS5 auth-based circuit separation when no peers exist
- **OPFS keypair persistence** — `.onion` address survives page reloads via Origin Private File System
- **Web Locks** — Prevents multiple tabs from fighting over the HS listener port
- **Ed25519 via Web Crypto** — `.onion` v3 address derivation (SHA3-256 checksum) and client auth signatures
- **Visibility-aware animations** — Canvas loops pause when tab is hidden to save battery
- **O(n) buffer handling** — Chunk-then-concat pattern for SOCKS5/HTTP response reading

### Browser API surface

| API | Usage |
|-----|-------|
| Direct Sockets | `TCPSocket` (SOCKS5, SOCKS5+auth), `TCPServerSocket` (HS listener + OHTTP relay endpoint) |
| WebMCP | `navigator.modelContext.registerTool()` x 7 |
| Web Crypto | ECDH P-256 (OHTTP relay), Ed25519 (keygen/sign/verify), AES-GCM (envelope encryption), SHA-256 |
| OPFS | `navigator.storage.getDirectory()` for keypair persistence |
| Web Locks | `navigator.locks.request()` for HS singleton |
| File System Access | `showDirectoryPicker()` for Tor data dir |
| Service Worker | Offline asset caching |
| Trusted Types | CSP compliance (IWA requirement) |

### What an AI agent can do with these tools

```
Agent: "Set up OHTTP privacy"
→ manageOHTTPRelay({ action: "volunteer" })
→ { volunteering: true, pubkey: "BFx9a...", localOnion: "abc...onion" }

Agent: "Add a relay peer I know about"
→ manageOHTTPRelay({ action: "addPeer", peerOnion: "xyz...onion", peerPubkey: "BGh2..." })
→ { success: true, totalPeers: 1, note: "fetchOnion with useOHTTP=true will now route through available peers." }

Agent: "Fetch a .onion site with OHTTP privacy"
→ fetchOnion({ url: "http://duckduck...onion/", useOHTTP: true })
→ { success: true, status: 200, ohttpMode: "peer-relay", ohttpRelay: "xyz...onion" }

Agent: "No peers available — what happens?"
→ fetchOnion({ url: "http://example...onion/", useOHTTP: true })
→ { success: true, status: 200, ohttpMode: "circuit-isolation" }

Agent: "Connect to another agent's hidden service"
→ holepunch({ action: "connect", targetOnion: "abc123...xyz.onion", port: 80 })
→ { success: true, sessionId: "...", status: "connected" }

Agent: "Lock down my hidden service"
→ manageTrustedClients({ action: "add", clientId: "agent-b", pubkey: "base64..." })
→ { success: true, note: "Access control is now ACTIVE" }
```
